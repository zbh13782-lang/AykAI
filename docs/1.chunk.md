```Python
def _detect_chunk_mode(source: str, metadata: dict[str, Any]) -> str:  
    """检查格式"""  
    normalized_source = (source or "").lower()  
    mime_type = str(metadata.get("mime_type", "")).lower()  
    file_type = str(metadata.get("file_type", "")).lower()  
  
    if normalized_source.endswith((".md", ".markdown")) or mime_type == "text/markdown" or file_type == "markdown":  
        return "markdown"  
    if normalized_source.endswith(".pdf") or mime_type == "application/pdf" or file_type == "pdf":  
        return "pdf"  
    return "text"
```

#### 它的处理逻辑：

1. **空行跳过**
2. **代码块识别**
    - 看到以 `开头的行，就一直收集到闭合` 为止
3. **表格识别**
    - 如果一行包含 `|`，就尝试把连续的表格行收集起来
    - 至少两行才算表格
4. **列表识别**
    - 识别 `-` 、`*` 、`+` 、`1.` 这种列表项
    - 连续列表会被合并成一个块
5. **普通段落**
    - 其他内容按段落收集，直到空行或遇到新的结构
 
这样做是为了让 Markdown 不只是“按字符长度机械切块”，而是尽量保留结构语义

## Q & A
### 1. 为什么需要 chunking？
为什么不能整篇文档都给llm ？
chunk size 太大，太小有什么影响？
overlap作用？你是怎么确定这两个参数的？- 你怎么通过实验选择参数？


chunking 的作用是把长文档拆成更小的语义单元，可以提升检索力度，保留上下文片段，如果不做 chunking，整篇文档太长，embedding 效果会变差，检索命中也不精准。

chunk_size：
 太大：
- 一个 chunk 里包含很多无关内容
- 检索出来噪声大
- 召回后喂给模型时上下文冗长
- 语义聚合不够精细

太小：
- 一个概念可能被切碎
- 上下文不完整
- 检索虽然更精准，但答案生成可能缺信息
- 过多碎片会增加索引数量和检索成本

所以 chunk size 本质上是在平衡**召回精度**和**语义完整性**。

参数确定：
1. 先定一个基础值
2. 用真实问答集测试
3. 看 recall@k、MRR、答案准确率
4. 再调整 chunk size 和 overlap

### 2.你会怎么设计 chunking 策略？
Markdown、PDF、代码、网页分别怎么切？
结构化文档和非结构化文档的策略有什么不同？
PDF 一般先按页切，再结合段落和自然语言边界切。

因为 PDF 常常来自版面抽取，天然结构比 Markdown 弱，所以要更依赖：

- 页码
- 段落
- 换行
- 句号
- 空格

如果是扫描件 OCR 文本，还要额外注意：

- 断行
- 乱码
- 页眉页脚重复
- 表格错位


### 3. 如何评估 chunking 效果？
你怎么判断你的 chunking 做得好不好？
评估指标有哪些？
是看召回率、准确率、MRR，还是人工评估？
你怎么做 A/B test？


通常从三层评估：

####  1）检索层

- recall@k
- precision@k
- MRR
- nDCG

#### 2）生成层

- 答案正确率
- 是否引用到正确上下文
- 幻觉率是否下降

#### 3）人工评估

- chunk 是否完整
- 语义是否被切断
- 检索结果是否可解释

最关键的是用真实问题集做 A/B test。


### 4. 实际工程问题

如何给每个 chunk 建 ID？ 
**唯一、可追踪、可关联**
如何保留 parent-child 关系？
**child 要存 parent_id，parent 要存 child_ids**。
如何支持增量更新？ 
**chunk ID 和文档版本要稳定**，不能每次全量重建
- 对 chunk 内容做 hash
- 内容没变就不重建
- 内容变了才更新
如何处理重复内容？

重复内容主要来自：
- 页眉页脚
- 导航栏
- 模板文本
- 表格重复行
- 文档里大量相似段落

 处理方法：
 1）预清洗

在 chunking 前先去掉：

- 页眉页脚
- 重复免责声明
- 导航菜单
- 空行和无意义符号

 2）去重

对 chunk 做 hash，如果内容完全一样，就只保留一份。

 3）近似去重

对于高度相似内容，可以用：

- SimHash
- MinHash
- embedding 相似度阈值

 4）检索后去重

即使库里有重复 chunk，也可以在召回后做去重，避免 top-k 全是相似结果。


如何处理超长文档？

如果 section 很短，还要不要继续切？

一般**不需要继续切**，短 section 直接保留更好。
只有在短 section 里还混有多个子主题，那就切

如果 chunk 太短，为什么直接丢弃？
太短没什么内容，不要了
宁可少保留一点碎片，也不要把大量无意义短块送进向量库。
