## 第二步：Embedding

这一篇只讲 chunk 之后最关键的一段：Embedding 与索引落库。

目标不是“把文本变向量”这么简单，而是把 chunk 变成三类可检索资产：

1. 子块向量索引（Milvus）
2. 父块语义索引（PostgreSQL）
3. 子块关键词索引（Elasticsearch）

---

## 1. Embedding 这一层到底做了什么

### 1.1 输入与输出

ingest graph 在 chunk 完成后会进入 embedding 节点，核心输入输出如下：

- 输入：`children[].content`
- 输出：`child_embeddings[]`

每个 child chunk 都会得到一个同维度向量，向量维度由 embedding 模型决定，后续 Milvus collection 的 schema 也必须与之严格一致。

### 1.2 为什么只对 child 做 embedding

父子分层后，child 是检索粒度，parent 是生成上下文粒度。

- child 更短，向量表达更聚焦，召回精度更高
- parent 更长，直接向量化会引入语义混叠，且成本更高
- 工程上采用“child 召回 + parent 回填”通常更稳

### 1.3 Embedding 阶段的工程约束

这一步要重点关注三个一致性：

- 顺序一致：`children[i]` 必须和 `child_embeddings[i]` 对齐
- 维度一致：模型输出维度必须与向量库字段定义一致
- 批量一致：批次失败重试后不能发生重复写或错位写

如果这三点没有守住，后续检索会出现“召回到了不相关片段但看不出原因”的隐性错误。

---

## 2. 父子索引设计（重点）

### 2.1 为什么做父子索引

RAG 的核心矛盾是：

- 检索要细粒度（更容易命中）
- 生成要高完整度（避免断章取义）

父子索引就是用两种粒度同时解决这个矛盾。

### 2.2 数据组织方式

典型结构如下：

- `parent_id`：父块主键，贯穿全链路
- `child_id`：子块主键
- `doc_id/source`：文档身份信息
- `offset/range`：子块在父块或原文中的位置

关系是：一个 parent 对应多个 child，child 必须带 `parent_id` 外键信息。

### 2.3 三库分工（本项目）

- PostgreSQL：存 parent（完整语义段、元信息、可回填内容）
- Milvus：存 child 向量（向量检索主路径）
- Elasticsearch：存 child 文本与检索字段（BM25 主路径）



### 2.4 写入顺序建议

项目里推荐把写入视作一个逻辑事务，顺序可按以下思路：

1. 先写 parent（确保回填源存在）
2. 再写 child vector（Milvus）
3. 再写 child lexical（Elasticsearch）

若任一步失败，进入重试与 repair queue，保证最终一致性

### 2.5 召回阶段如何用父子索引

检索命中的是 child（向量或关键词），但最终给模型的上下文通常用 parent：

1. 从命中的 child 中拿到 `parent_id`
2. 批量读取 parent 内容
3. 组装 references（包含 child 命中证据 + parent 上下文）

这一步是“检索命中率”和“答案可读性”之间的桥梁。

## 3. Elasticsearch 存储细节（重点）

### 3.1 ES 在这套架构中的角色

ES 不是向量库替代，而是 lexical 召回引擎：

- 负责关键词、实体名、术语的精确命中
- 与向量检索互补，提升整体召回稳定性

### 3.2 索引字段建议

child 文档建议至少包含：

- `chunk_id`
- `parent_id`
- `doc_id`
- `content`
- `source`
- `created_at` 或版本字段

这样能支持：检索、回填、追踪、增量更新。

### 3.3 分词与 analyzer 策略

当前项目支持 IK 分词插件，同时包含降级策略：

- 有 IK：使用 `ik_smart/ik_max_word` 配置中文分词
- 无 IK：fallback 到标准 analyzer，避免索引初始化失败

这一点非常关键，能保证不同部署环境下服务可启动。

### 3.4 upsert 与幂等

ES 写入最好按 `chunk_id` 幂等 upsert：

- 重复 ingest 同一 chunk 不应产生脏重复
- 文档更新要可覆盖旧版本
- 批量写入时注意 batch size，避免内存与请求体过大

### 3.5 ES 不可用时的降级

本项目具备 BM25 fallback：

- ES 初始化失败时不阻断主服务启动
- 关键词检索走本地倒排实现
- 质量可能下降，但可用性优先

## 4. Embedding + 父子索引

可以把第二步理解为三句话：

1. Embedding 解决“语义怎么找”
2. 父子索引解决“找得准还要讲得全”
3. ES 解决“关键词和实体怎么稳命中”

## Q & A

### 1）为什么只对 child 做 embedding，不直接对 parent 做？

回答要点：

- child 粒度更利于高精度召回
- parent 过长会语义混叠、成本更高
- 实践上用 child 命中 + parent 回填效果更稳

### 2）父子索引里 `parent_id` 的作用是什么？

回答要点：

- 连接召回证据与生成上下文
- 支持检索后快速回填父块
- 支持引用追踪与可解释性

### 3）ES 在混合检索中不可替代的价值是什么？

回答要点：

- 对实体词、专有名词、精确短语更敏感
- 语义检索难以稳定覆盖 exact term
- 与向量检索互补，提高召回鲁棒性

### 4）IK 不可用时为什么要做 analyzer fallback？

回答要点：

- 避免索引初始化失败导致服务不可用
- 不同环境插件安装状态不一致
- 先保证系统可运行，再优化检索质量

### 5）如何保证 embedding 写入链路的一致性？

回答要点：

- child 与向量按索引严格对齐
- 维度校验与 schema 校验前置
- 批量 upsert 采用幂等键与失败重试

### 6）如果线上出现“召回结果看起来随机”，优先排查什么？

回答要点：

- embedding 维度与 Milvus schema 是否一致
- child/embedding 顺序是否错位
- parent_id 映射是否断裂
- ES analyzer 与 mapping 是否与预期一致
